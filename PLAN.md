# План разработки `ozon-ord`

## Цель

Сделать Python-скрипт, который:

1. Подключается к Google Sheets по заданной таблице.
2. Забирает из неё данные.
3. Преобразует их в валидную внутреннюю структуру.
4. Отправляет подготовленные данные в API ОЗОН ОРД.

## Базовые допущения

- Таблица Google Sheets либо публично доступна на чтение, либо для неё можно настроить доступ через service account.
- У нас будет документация или примеры запросов по API ОЗОН ОРД.
- В таблице есть фиксированная структура колонок, которую можно считать контрактом.

## Этап 1. Анализ входных данных из Google Sheets

Цель этапа: понять формат таблицы и зафиксировать схему данных, с которой будет работать код.

Задачи:

1. Определить способ доступа к таблице:
   - публичный экспорт CSV/TSV;
   - Google Sheets API;
   - `gspread` + service account.
2. Считать первую вкладку и получить:
   - список колонок;
   - пример 5-10 строк;
   - наличие пустых/служебных колонок;
   - потенциальные обязательные поля.
3. Зафиксировать схему строки:
   - имя поля;
   - тип значения;
   - обязательно ли поле;
   - как поле должно попасть в ОЗОН ОРД.
4. Отдельно определить проблемные места:
   - пустые строки;
   - дубли;
   - даты в разных форматах;
   - числа, суммы, ИНН/ID в строковом виде;
   - объединённые ячейки, комментарии, формулы.

Результат этапа:

- понятная схема входных данных;
- решение по способу подключения;
- список правил нормализации данных.

## Этап 2. Реализовать модуль парсинга Google Sheets

Цель этапа: получить стабильный парсер, который возвращает чистые Python-структуры.

Задачи:

1. Подготовить модуль, например:
   - `src/sheets_reader.py` или `sheets_reader.py`;
   - отдельный конфиг для URL таблицы и параметров доступа.
2. Реализовать загрузку данных из таблицы.
3. Преобразовать строки в нормализованную модель:
   - trim строк;
   - приведение пустых значений к `None`;
   - парсинг дат;
   - парсинг чисел;
   - унификация названий полей.
4. Добавить первичную валидацию:
   - обязательные поля;
   - корректность форматов;
   - сообщения об ошибках по строкам.
5. Сделать CLI/entrypoint для локального запуска:
   - загрузить таблицу;
   - вывести количество строк;
   - вывести 1-2 нормализованные записи;
   - сохранить ошибки в лог.

Результат этапа:

- рабочий скрипт чтения Google Sheets;
- список нормализованных записей;
- отчёт по невалидным строкам.

## Этап 3. Спроектировать внутреннюю модель данных

Цель этапа: отделить формат таблицы от формата API ОЗОН ОРД.

Задачи:

1. Описать внутренние сущности через `dataclass` или `pydantic`.
2. Разделить:
   - сырой формат строки из таблицы;
   - нормализованную бизнес-модель;
   - payload для API ОЗОН ОРД.
3. Подготовить маппинг полей таблицы в поля ОЗОН ОРД.

Результат этапа:

- прозрачная схема преобразования `Google Sheets -> внутренняя модель -> API payload`.

### Зафиксированные правила маппинга в ОЗОН ОРД

- `Креатив` <- `creative_id`
- `Площадка` <- `channel_url`
- `Тип рекламной кампании` <- константа `Иное`
- `Показы фактические` <- `reach`
- `Показы плановые` <- `reach`
- `Даты показов фактические` <- диапазон `publication_date .. publication_date + 1 день`
- `Даты показов плановые` <- диапазон `publication_date .. publication_date + 1 день`
- `Ставка НДС` <- константа `Без НДС`
- Режим суммы <- всегда `без НДС`
- `Сумма оказанных услуг` <- `price_with_tax`
- `Стоимость ед. оказания услуг с НДС` <- `price_with_tax`

### Что нужно проверить перед интеграцией API

- Не конфликтует ли правило `режим суммы = без НДС` с передачей значения из колонки `цена с налогом`.
- В API дата показов передаётся одной датой, диапазоном или отдельными полями начала/окончания.
- Нужно ли отдельно передавать вычисленные поля `сумма без НДС` и `сумма НДС`, или интерфейс/сервер считает их автоматически.

## Этап 4. Интеграция с API ОЗОН ОРД

Цель этапа: отправлять валидные записи в ОЗОН ОРД.

Задачи:

1. Определить способ авторизации в API ОЗОН ОРД.
2. Подготовить клиент API:
   - базовый URL;
   - заголовки;
   - таймауты;
   - retry на временные ошибки.
3. Реализовать отправку одной записи.
4. Реализовать пакетную обработку:
   - dry-run режим;
   - отправка по одной записи или батчами;
   - логирование успешных и неуспешных отправок.
5. Обработать ошибки API:
   - 4xx как ошибки данных;
   - 5xx как временные ошибки;
   - повторные попытки там, где это безопасно.

Результат этапа:

- рабочая интеграция с ОЗОН ОРД;
- предсказуемое поведение при ошибках.

## Этап 5. Идемпотентность и контроль повторных загрузок

Цель этапа: не создавать дубли при повторном запуске.

Задачи:

1. Определить уникальный ключ записи.
2. Решить, где хранить статус обработки:
   - локальный файл;
   - SQLite;
   - отдельная служебная вкладка;
   - внешний storage.
3. Добавить режимы:
   - `preview`;
   - `sync`;
   - `retry-failed`.

Результат этапа:

- безопасный повторный запуск без ручной сверки.

## Этап 6. Тесты и эксплуатация

Задачи:

1. Добавить unit-тесты на:
   - парсинг строк;
   - нормализацию;
   - валидацию;
   - маппинг в payload.
2. Добавить фикстуры с примером данных таблицы.
3. Описать в `README.md`:
   - как настроить доступ к Google Sheets;
   - какие переменные окружения нужны;
   - как запускать dry-run;
   - как запускать боевую отправку.

Результат этапа:

- проект можно запускать и поддерживать без ручного разбора кода.

## Порядок работы

1. Разобрать структуру Google Sheets и подтвердить способ доступа.
2. Реализовать парсер и нормализацию данных.
3. Зафиксировать внутреннюю модель.
4. Подключить API ОЗОН ОРД.
5. Добавить защиту от дублей, тесты и документацию.

## Что делаем прямо сейчас

Первый рабочий спринт:

1. Проверяем, как читать эту конкретную Google Sheets.
2. Получаем заголовки и примеры строк.
3. Пишем минимальный парсер, который возвращает нормализованный список записей.
4. Фиксируем контракт данных для следующего этапа интеграции с API ОЗОН ОРД.
